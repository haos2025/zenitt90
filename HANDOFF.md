# Zenith OTT — HANDOFF (читать первым)

Этот документ — точка входа для любой модели/человека, продолжающих работу
над проектом Zenith OTT после этой сессии. Порядок чтения:

1. **`HANDOFF.md`** (этот файл) — роль, правила, протокол возврата
2. **`PROJECT_OVERVIEW.md`** — что за проект, из чего состоит, текущий статус
3. **`ARCHITECTURE_DECISIONS.md`** — что и почему решено (не переоткрывать
   заново без причины)
4. **`KNOWN_PITFALLS.md`** — реальные классы ошибок, уже дважды-трижды
   наступленные за историю проекта — самое важное для того, кто будет
   исправлять баги
5. **`CHECKLIST.md`** — построчный журнал каждого применённого фикса
6. **`NEXT_TASKS_PROMPT.md`** — открытые задачи с готовыми спецификациями

## Контекст: как устроена работа над этим проектом

Автор проекта использует несколько ИИ-инструментов в разных ролях: Claude —
основной архитектор (ревью кода, спецификации, финальная проверка), другие
модели — для аудита и большого объёма правок, когда их много одновременно.
**Эта сессия — именно такой заход**: множество ошибок, накопившихся в
проекте, нужно исправить, и это делается в другом инструменте. После —
автор возвращается к Claude с результатом.

**Из этого следует:** твоя роль сейчас — исполнитель конкретного объёма
работы (исправить ошибки), не архитектор, принимающий решения в одиночку.
Если по ходу исправления возникает архитектурный вопрос (не однозначный
баг, а выбор между вариантами) — задать его человеку явно, не решать молча.
Именно так был построен весь процесс в предыдущих сессиях с Claude — см.
`ARCHITECTURE_DECISIONS.md`, там нет ни одного решения, принятого моделью
без участия автора.

## Правила, которые нужно соблюдать

1. **Два раздельных репозитория.** `zenitt90-main` (Android, Kotlin) и
   `zenith-backend` (Python/FastAPI) — не один проект, связаны только по
   HTTP. Правки в одном не должны молча предполагать одновременные правки в
   другом без явного упоминания — если фикс на одной стороне требует
   изменений контракта на другой, это нужно проговорить отдельно.

2. **Не гадать версии зависимостей и API незнакомых библиотек.** Эта
   кодовая база уже дважды страдала от несуществующих версий пакетов и
   непроверенных предположений об API (см. `KNOWN_PITFALLS.md`, разделы 5-6).
   Любая версия пакета или сигнатура метода незнакомой библиотеки — сначала
   искать подтверждение (документация, PyPI/Maven), потом использовать. Не
   найдено подтверждение — не гадать, использовать то, что уже точно
   работает в проекте.

3. **"Задокументировано как рабочее" ≠ "реально подключено".** Самый частый
   класс багов в этом проекте — полностью реализованная функция без единого
   вызывающего места. При любом баге вида "штука вроде должна работать, но
   не работает" — сначала проверить `grep -rn` на реальные call site'ы,
   не доверять комментариям/докстрингам на слово.

4. **YouTube-плеер не трогать.** Сознательно снято с повестки автором
   проекта (embed-ссылка вместо прямого файла) — не пытаться "починить
   заодно", если попадётся на глаза.

5. **Не переоткрывать решённые архитектурные вопросы** без явного запроса
   человека — они не случайны, у каждого есть обоснование в
   `ARCHITECTURE_DECISIONS.md`. Если решение выглядит подозрительно или
   создаёт проблему при исправлении бага — сообщить об этом прямо, не
   менять втихую.

6. **Документировать так же, как в `CHECKLIST.md`.** Формат: что было не
   так → почему → diff/суть фикса → что это меняет. Не просто "исправлено",
   а объяснение, воспроизводимое без контекста разговора. Это то, что
   позволяет любой следующей сессии (включая возврат к Claude) понимать
   историю без необходимости заново объяснять на словах.

## Протокол возврата к Claude

Когда исправления в другом инструменте закончены, для эффективного ревью
Claude нужно получить (в порядке убывания важности):

1. **Список изменённых файлов с путями** — по обоим репозиториям отдельно
2. **Что было не так и что изменено** — по каждому файлу или логической
   группе правок, не обязательно длинно, но по существу
3. **Логи реальных ошибок**, если что-то проверялось вживую (GitHub Actions,
   Render deploy log, логи с реального устройства) — сырой текст лога, не
   пересказ своими словами
4. **Открытые вопросы**, если по ходу исправления встретился архитектурный
   выбор, который не был очевиден — что рассматривалось, что выбрано и
   почему, или явно "не решил, нужно мнение"

Claude в свою очередь: сверит правки на соответствие уже установленным
паттернам проекта (`KNOWN_PITFALLS.md`), обновит `CHECKLIST.md`/
`NEXT_TASKS_PROMPT.md`/`ARCHITECTURE_DECISIONS.md` актуальным состоянием,
и продолжит с того места, где остановились открытые задачи.
