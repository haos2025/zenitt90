# Zenith OTT — сводный документ для продолжения в новом чате

Это техническая сводка всей истории разработки Android TV/мобильного IPTV-приложения
"Zenith OTT" (пакет `com.platinum.ott`), собранная из очень длинного чата с Claude.
Разработка велась через **Python-скрипт-генератор** (`setup_zenith_v2_combined.py`),
который создаёт весь проект одним запуском — так удобнее передавать код между разными
ИИ и итерациями, чем десятками отдельных файлов.

Если ты (следующий ИИ/чат) продолжаешь эту работу — прочитай этот документ целиком
перед тем, как что-то менять. Многие решения здесь приняты НЕ произвольно, а по
результатам реальных ошибок компиляции — не переоткрывай уже закрытые вопросы
без веской причины.

## 1. Контекст и рабочий процесс

- Автор — Shadow, разработчик-одиночка, Windows 10, публикует код через **GitHub Desktop**
  (не консольный git), собирает **только через GitHub Actions** — Android Studio
  и Gradle локально не установлены.
- Используется мульти-ИИ workflow: Claude как основной архитектор, плюс GPT, Gemini,
  DeepSeek, "Mimo" — для аудита и параллельной разработки. Их рекомендации/аудиты
  **регулярно оказывались неполными или ошибочными** при реальной проверке — это
  подтверждённый паттерн, не предвзятость. Всегда перепроверяй код фактически
  (compile-check, grep, чтение реального содержимого файлов), а не верь табличкам
  "✅ 0 ошибок" на слово.
- У среды разработки (песочница Claude) **нет доступа к Android SDK / реальному Gradle**
  — весь код никогда не компилировался напрямую самим ассистентом. Первая реальная
  компиляция происходит только в GitHub Actions на машине Shadow. Из-за этого баги
  вскрывались только по логам, присылаемым пользователем — итеративно, раунд за раундом.

## 2. Что уже построено (по состоянию на конец этого чата)

### Источник кодовой базы

Слияние из:
1. 10 python-скриптов v1.9 (fixed + p1..p9) — полный функционал: auth M3U/Xtream,
   Room-кэш, OTA JS-парсеры (QuickJS), детальный экран, выбор качества, кастомный
   плеер, подпись APK+CI.
2. PowerShell-скрипт v2.0 — давал корректный Gradle-каркас (settings.gradle.kts,
   корневой build.gradle.kts), которого в v1.9 не было вообще.
3. 14+ раундов правок поверх слияния (см. раздел 4).

### Архитектура

- **DI**: `ServiceLocator` (объект-синглтон), НЕ Hilt. Осознанное решение — см. раздел 5.
- **Слои**: `data/local` (Room), `data/remote` (Retrofit), `domain/usecase`,
  `presentation/screens/*` (Compose), `core/` (ServiceLocator, preferences, JS engine).
- **UI**: Jetpack Compose + `androidx.tv.material3` (Android TV Compose, stable).
- **Плеер**: ExoPlayer (Media3) с явной поддержкой HLS/DASH.
- **Сеть**: Retrofit + OkHttp + RetryInterceptor (exponential backoff + jitter).
- Финальный файл проекта: ~90 файлов (после 14 раундов); есть параллельная
  веб-версия "v3" от Mimo на 137 файлов с доп. фичами (см. раздел 6).

### Итоговый стек версий (актуальный, проверенный компилятором)

```
AGP            = 8.6.0
Kotlin         = 2.1.20   (НЕ 1.9.24! см. раздел 4, раунд 13)
KSP            = 2.1.20-2.0.1
Room           = 2.7.1    (НЕ 2.6.1! см. раздел 4, раунд 14)
Compose BOM    = 2024.09.00
tv-foundation  = 1.0.0 (stable)
tv-material    = 1.1.0 (stable)
Media3         = 1.4.0 (+ media3-exoplayer-hls, media3-exoplayer-dash)

```

Compose-компилятор подключается плагином `org.jetbrains.kotlin.plugin.compose`
(способ для Kotlin 2.x), а НЕ через `composeOptions.kotlinCompilerExtensionVersion`
(старый способ для Kotlin ≤1.9 — для 2.x не работает вообще).

Все версии и их взаимная совместимость задокументированы в файле проекта
`COMPATIBILITY.md` — там же источники (gradle.org, androidx release notes).

## 3. CI/CD — итоговая, финально отработанная схема

**Ключевой урок всей многораундовой истории**: CI должен быть изолирован от
самого ненадёжного звена — ручной загрузки бинарного `gradle-wrapper.jar`.

- **Локально** (`gradlew`, `gradlew.bat`, `gradle-wrapper.jar`) — генерируются
  скриптом, коммитятся в git. Полезно для локальной разработки / открытия в
  Android Studio (если Shadow когда-нибудь её поставит).
- **В CI** (`.github/workflows/build.yml`, `release.yml`) — Gradle ставится
  напрямую через `gradle/actions/setup-gradle@v4` (`gradle-version: '8.7'`),
  все команды идут через `gradle ...`, НЕ `./gradlew ...`. Если committed jar
  битый/отсутствует — это НЕ блокирует сборку.
- `gradle/actions/wrapper-validation@v4` оставлена, но с `continue-on-error: true`
  — информационная проверка, не блокер.
- `debug.keystore` — реальный (сгенерирован `keytool`, не выдуман), коммитится
  в репозиторий (пароль "android" — публичный дефолт AGP, не секрет). Даёт
  одинаковую debug-подпись на любой машине и в CI.
- Release-подпись — через GitHub Secrets (`KEYSTORE_BASE64`, `KEY_ALIAS`,
  `KEY_PASSWORD`, `STORE_PASSWORD`), инструкция в `README_SIGNING.md` внутри
  проекта.
- Dependabot настроен с группировкой: `org.jetbrains.kotlin*` +
  `com.google.devtools.ksp*` + `androidx.compose*` — в одну группу, чтобы
  никогда не бампались независимо (см. раздел 4, раунд 11 — именно это
  один раз уже сломало сборку).

### ⚠️ НЕЗАКРЫТЫЙ баг на момент написания документа

В параллельной ветке Mimo (`setup_zenith_v3_final.py`) обнаружен реальный баг:
в `release.yml` шаг `assembleRelease` вызывается БЕЗ флагов подписи
(`-Pandroid.injected.signing.*`), в отличие от соседнего `bundleRelease`,
у которого эти флаги есть. Из-за этого APK (не AAB) в релизе либо не соберётся,
либо будет неподписанным. **Нужно добавить те же `-P` флаги в `assembleRelease`.**
Это не исправлено в финальной версии на момент конца чата — сделай это первым
делом, если продолжаешь с v3-веткой.

### Важное наблюдение про паттерн ошибок

Практически все реальные (не мнимые) баги делятся на 2 категории:
1. **Версионная несовместимость** экосистемы Kotlin/KSP/Compose/Room/AGP —
   решается ТОЛЬКО реальной компиляцией, никакое статическое чтение кода
   этого не поймает.
2. **Configuration-cache несовместимые вызовы** (`project.exec`, `System.getenv`
   напрямую в `build.gradle.kts`) — решается заменой на Provider API
   (`providers.exec`, `providers.environmentVariable`).

Если увидишь новую ошибку сборки — сначала проверь, не подходит ли она под
одну из этих двух категорий.

## 5. Ключевые архитектурные решения и ПОЧЕМУ (не переоткрывать без причины)

### ServiceLocator, а не Hilt

Рассматривалась параллельная ветка "Zenith Enterprise Generator" (PDF от другого
ИИ) на Hilt DI. Решение: **не переходить полностью на Hilt**. Причины:
- Enterprise-ветка — всего 33 файла инфраструктуры, БЕЗ единого экрана,
  без OTA-логики, без плеера. Полный переход = выбросить весь рабочий функционал
  ради недоделанного каркаса.
- В ней найден confirmed баг: `AppModule.kt` использует `DynamicBaseUrlInterceptor`,
  которого нигде не определено — не скомпилируется.
- В ней же — фейковый SSL-pin (`sha256/AAAA...=`, все нули/A) — опасная заглушка.
- Из неё **взяты только точечные паттерны** (не архитектура целиком):
  `RetryInterceptor.kt` (качественная реализация, exponential backoff+jitter) —
  перенесён без изменений в наш `data/remote/`.

### Kotlin 2.x — не "модернизация ради моды"

В раунде 4 было решено НЕ бампать Kotlin с 1.9.24 на 2.x без доказательств,
что это нужно (осторожность). В раунде 13 нашлось прямое доказательство
(несовместимость байткода с tv-material/tv-foundation) — тогда бамп сделан.
Мораль: не бампай мажорные версии экосистемы превентивно, но и не бойся
делать это, когда есть конкретное доказательство необходимости.

### debug.keystore коммитится в репозиторий

Пароль `"android"`, alias `"androiddebugkey"` — это публичный, задокументированный
дефолт самого AGP, не секрет. Коммитить его нужно, чтобы debug-подпись была
одинаковой на любой машине и в CI (иначе `adb install` требует переустановки
при каждом переключении между локальной сборкой и CI-сборкой).

## 7. Как проверять код на этом проекте (методология, которая реально работала)

Поскольку в среде разработки нет доступа к Android SDK/Gradle, весь код
никогда не компилировался напрямую. Методология проверки, которая сработала:

1. **Статический аудит через Python-скрипты**: распаковать `project_files`
   dict из `.py`-генератора в реальные файлы на диске, затем:
   - Проверить баланс скобок `{`/`}` во всех `.kt`
   - Проверить версии в `libs.versions.toml` на взаимную согласованность
     (особенно Kotlin/KSP — версия KSP ДОЛЖНА начинаться с версии Kotlin,
     формат `<kotlin>-<ksp-patch>`)
   - Кросс-проверить импорты `import com.platinum.ott.*` против реально
     объявленных классов/функций/свойств (учитывая top-level `fun`,
     `@Composable fun`, extension-функции — не только `class`/`object`)
   - Искать паттерн `SurfaceDefaults.colors` возле `onClick =` (баг tv-material)
   - Искать `suspend fun` с `@Query("DELETE...")`/`@Query("UPDATE...")` без
     явного типа возврата (баг KSP2 `google/ksp#2957`)
   - Искать `project.exec`/`System.getenv` напрямую в `.gradle.kts` (баг
     configuration-cache)
2. **web_search для проверки версий/URL** — НЕ доверять памяти/тренировочным
   данным для текущих версий пакетов, официальных URL дистрибутивов, чексумм.
   Дважды за этот чат неверный URL (`services.gradle.org/.../wrapper.jar`)
   был "подтверждён" неправильно логической догадкой вместо прямой проверки.
3. **Реальные логи компиляции от пользователя — единственный 100% надёжный
   источник истины.** Все теоретические/статические проверки — это способ
   ПОДГОТОВИТЬСЯ к реальной ошибке, не замена ей. Когда приходит реальный лог
   — читай его буквально, не додумывай, ищи точную причину через web_search
   если незнакома (как в раунде 13 и 14 — оба раза причина находилась через
   поиск конкретного текста ошибки).

## 9. Что делать дальше (открытые задачи)

1. Починить `assembleRelease` в `release.yml` v3-ветки (флаги подписи).
2. Решить: продолжать с v2 (90 файлов, полностью проверено) или влить
   исправленную v3 (137 файлов, новый функционал, частично проверено)?
3. Если сборка в GitHub Actions пройдёт полностью зелёной — это будет ПЕРВЫЙ
   раз за всю историю проекта. До этого момента считать код "вероятно
   рабочим", не "гарантированно рабочим" — слишком много раундов показали
   расхождение между тем, что выглядит правильным, и тем, что реально
   компилируется.
4. Не полагаться на Dependabot автослияние без проверки — Kotlin/KSP/Compose
   версии жёстко связаны, механизм группировки уже настроен, но человек
   всё равно должен смотреть, что мержит.
